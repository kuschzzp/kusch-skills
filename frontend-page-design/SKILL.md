# 前端设计规范与美化

用于两类任务：`从0生成页面` 与 `已有页面美化`。始终先确认用户偏好，再按模式执行。

## 模式选择

1. 若用户没有提供现有页面代码，按 `模式 A：从0生成页面` 执行。
2. 若用户提供了现有页面代码或明确要求改造，按 `模式 B：已有页面美化` 执行。

## 通用前置步骤（两种模式都必做）

1. 先询问并确认用户视觉偏好。
- 先不要直接改样式，先确认用户偏好与边界。
- 读取 `references/style-preferences-questionnaire.md`，至少确认：配色方向、动效强度、是否使用毛玻璃、是否使用阴影/拟物/渐变等风格。
- 读取 `references/color-palettes.md`，提供 3-5 套候选色板供用户选择或微调。

2. 确认技术与约束。
- 确认技术栈（HTML/CSS、React、Vue 等）与是否可引入依赖。
- 确认是否有品牌规范、设计系统、无障碍要求、性能要求。

## 模式 A：从0生成页面

1. 先做页面目标定义与信息架构。
- 明确页面目标、用户类型、核心转化动作（CTA）。
- 规划页面区块与内容优先级（首屏、价值点、功能说明、社会证明、FAQ、页脚等）。
- 读取 `references/page-generation-playbook.md`，按“结构先行，再做视觉”执行。

2. 先搭结构再做视觉系统。
- 先产出语义化结构和响应式布局骨架。
- 再建立 token（颜色、间距、字号、圆角、阴影、动效）。
- 最后细化组件外观与交互状态。

3. 完成质量校验。
- 验证桌面端与移动端可用性。
- 验证可访问性基线（语义化、焦点可见、对比度、可读性）。
- 验证核心路径和 CTA 行为符合目标。

## 模式 B：已有页面美化

1. 改代码前先做设计审查。
- 识别层级、间距节奏、字号体系、颜色使用、对齐、组件一致性、响应式断点等客观问题。
- 读取 `references/design-audit-checklist.md`，按关键维度打分并优先处理高收益问题。

2. 定义视觉方向和约束。
- 若项目已有设计系统，优先遵循现有规范。
- 大范围视觉改造前，先创建或统一间距、字号、圆角、阴影、颜色角色等 design token。
- 非必要不引入新库。

3. 按影响力从高到低重构结构与样式。
- 先修布局与信息层级（栅格、分区、分组、留白）。
- 再修字体与间距系统。
- 最后做组件细节 polish（按钮、卡片、表单、表格、导航状态）。
- 按页面区域读取 `references/polish-patterns.md` 选择具体策略。

4. 强化响应式与可访问性质量。
- 验证桌面端与移动端布局。
- 确保语义化结构、键盘焦点可见、对比度充足、文字可读。
- 动效保持克制且有目的；必要时遵循 reduced-motion 偏好。

5. 验证并输出结果。
- 确认没有功能回归。
- 说明做了什么、为什么体验更好、哪些项属于可选增强。

## 输出格式

返回结果时包含：

1. `任务模式`：从0生成 / 已有页面美化。
2. `用户偏好确认`：已确认的配色、风格、动效、毛玻璃等选择。
3. `结构与视觉方案`：页面结构决策或问题诊断结论。
4. `已实施改动`：具体布局/样式/组件修改项。
5. `质量检查`：响应式、可访问性、功能回归检查结果。
6. `可选下一轮`：非阻断增强建议。
